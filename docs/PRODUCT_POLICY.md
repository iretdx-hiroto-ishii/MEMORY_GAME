# プロダクト方針（AI・実装の基本方針）

> 画面設計書や Issue ごとに繰り返さない。**本ファイルが唯一の正本**とする。  
> 方針の変更はここだけを更新し、設計書・ルールは必要なら参照文のみ直す。

---

## 当面の優先順位

1. **なるはやリリース**し、**広告収入**を得ることを最優先とする。
2. 各画面・機能は **リリースに耐える最低限** に限定する（Must / Won't で切る）。

## AI・実装時の判断基準

| 判断 | 方針 |
|------|------|
| 機能追加の提案 | 収益 or リリース必須でないものは **Won't** または次イテレーション |
| スコープが曖昧なとき | 設計書の Must を満たせばマージ可。Should は明示指示があるまで着手しない |
| リファクタ・抽象化 | 今回の Must に直結するときのみ。見た目の整理だけは後回し |
| 依存パッケージ | 原則追加しない（広告 SDK 等は別タスクで明示） |
| 設計書 | 画面ごと `docs/design/<画面ID>.md`。本ファイルの方針は **コピーしない** |
| UI 共通 | 全画面 [`docs/design/UI_COMMON.md`](design/UI_COMMON.md)（`AppFooter`・コピーライト Must） |
| 実装タスク完了前 | [実装後の品質（セルフレビュー）](#実装後の品質セルフレビュー) を実施する |

## 実装後の品質（セルフレビュー）

コード変更を「完了」とする前に、**触った範囲**について以下を確認する。  
新たな不整合を生まないよう、**今回スコープ内の最小修正**のみ行う（無関係なリファクタ・抽象化はしない）。

### チェックリスト

| # | 観点 | 確認内容 |
|---|------|----------|
| 1 | **矛盾** | 設計書（`docs/design/`）の Must・AC と実装が一致しているか。二重定義・相反する state / props がないか |
| 2 | **重複** | 同じロジック・UI・定数のコピペが増えていないか。既存関数・コンポーネントを再利用できるか |
| 3 | **デッドコード** | 未使用の import・変数・関数・スタイル・到達不能分岐が残っていないか |
| 4 | **整合性** | 画面遷移・localStorage キー・命名が既存パターンと揃っているか。修正で別画面を壊していないか |
| 5 | **スコープ** | MVP 方針に反する機能追加や依存パッケージ追加をしていないか |

### 実施手順

1. 変更ファイルを一覧し、上記 1〜5 を確認する。
2. 問題があれば修正案を検討し、**必要最小限**だけ修正する。
3. 可能なら `npm run build` および `npm run lint` を実行し、エラーがあれば解消する。
4. 残した既知の課題（Should / Won't）があれば、完了報告に短く記載する。

### MVP 方針との両立

- セルフレビューは品質のための**ゲート**であり、スコープ拡大の口実にしない。
- 「ついで」の大規模整理は行わない。Must を満たす修正に留める。
